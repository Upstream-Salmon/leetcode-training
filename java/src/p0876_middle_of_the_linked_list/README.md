# 876. Middle of the Linked List

## 日付
2026-03-31

## 議論した内容
- **単方向リストの制約**: 
  - 一度末尾まで進むと逆方向には戻れない性質を確認。
  - 当初は「全要素を数えてから再度進む」2パス解法を検討したが、1回の走査（One-pass）で解く「Slow & Fast Pointers」への最適化を行った。
- **Javaのメモリモデルと参照**:
  - `ListNode current = head` としても、`current` 自体の代入（参照先の変更）であれば元の `head` は影響を受けないことを確認。
  - Javaの整数除算（`int / int`）が小数点以下を切り捨てる特性を利用し、中間地点の計算を簡略化できることを学んだ。
- **エッジケースの考慮**:
  - 要素数が偶数の場合、2つある真ん中の「後ろ側」を返すという条件を、`while` 文の終了条件でスマートに解決。
  - `NullPointerException (NPE)` を防ぐための短絡評価（Short-circuit evaluation）の重要性を確認。

## 計算量分析
- **時間計算量: O(n)**
  - リストを1回走査する間に解が得られる（実質的なステップ数は n/2）。
- **空間計算量: O(1)**
  - ポインタ用の変数（slow, fast）を2つ保持するだけであり、入力サイズに依存しない。

## 面接に向けた教訓
- **One-pass の推奨**: 
  - 大規模分散システムを扱う企業では、データの読み取り回数を減らすアルゴリズム（One-pass）が強く好まれる。
- **Two Pointers パターンの習得**: 
  - 今回の「ウサギとカメ」の手法は、Linked List の問題における「三種の神器」の一つ。中間地点の特定、サイクルの検出、後ろからN番目の特定などに即応用可能。
- **ガード節の徹底**:
  - `if (head == null)` のような例外的な入力を最初に弾く姿勢は、コードの堅牢性を示す重要な評価ポイント。
- **ガード句の要否**: 
  - 本アルゴリズムでは `while` 条件により `head == null` も自然にカバーされるが、実務や面接では「意図の明確化」と「不要な変数割当の回避」のために明示的なチェックが推奨される場合がある。
