# 217. Contains Duplicate (重複する要素)

## 課題概要
整数配列 `nums` が与えられたとき、配列内に同じ値が少なくとも2回出現する場合は `true` を返し、すべての要素が異なる場合は `false` を返します。

---

## 実装アプローチの比較

この問題に対して、計算量と特性の異なる3つのアプローチを実装しました。

| 解法 | 時間計算量 | 空間計算量 | 特徴 |
| :--- | :--- | :--- | :--- |
| **Brute Force** | $O(n^2)$ | $O(1)$ | 二重ループによる全探索。低速。 |
| **Sorting** | $O(n \log n)$ | $O(1)$ | 配列を並び替えて隣接比較。メモリ節約型。 |
| **HashSet** | **$O(n)$** | $O(n)$ | **最速のアプローチ。** 実務での標準。 |

---

## 各手法の解説

### 1. Brute Force (二重ループ)
全ての要素の組み合わせをしらみつぶしに比較します。
- **メリット:** 追加のデータ構造（メモリ）を一切使用しない。
- **デメリット:** 要素数 $n$ に対して計算量が指数関数的に増えるため、大規模データには不向き。

### 2. Sorting (ソート)
配列を昇順にソートし、同じ値が隣り合う性質を利用して判定します。
- **メリット:** `HashSet` を作成するメモリを節約できる。
- **デメリット:** `Arrays.sort()` によって元の配列の順序が書き換わる（破壊的変更）。また、ソート自体のコストがかかる。

### 3. HashSet (ハッシュセット)
Javaの `Set` インターフェースが持つ「重複を許さない」という特性を利用します。
- **メリット:** 配列を1回スキャンするだけで判定が完了するため、非常に高速。
- **テクニック:** `set.add(num)` の戻り値（既に存在すれば `false`）を直接 `if` 文の条件に使うことで、無駄な存在チェックを省いています。

---

## まとめ・考察
`Two Sum` では **HashMap** を使用して「値とインデックス」のペアを管理しましたが、今回の問題では「値が存在するかどうか」のみが焦点であるため、メモリ効率の観点から **HashSet** を選択するのがベストプラクティスです。

実務においては、処理速度を優先して **HashSet** を第一選択とし、メモリ制約が極めて厳しい環境（組み込み等）でのみ **Sorting** を検討するという使い分けが適切です。
